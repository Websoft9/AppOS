# App Lifecycle Source Build Spec JSON Draft

## Status
Proposed

## Context

AppOS already persists normalized lifecycle execution input in `app_operations.spec_json`.

Current implementation evidence shows that `spec_json` is the queue boundary contract and already carries normalized lifecycle-facing sections such as:

1. `metadata`
2. `exposure_intent`
3. source and adapter attribution
4. runtime-facing values such as `project_dir`

Source-based deployment now needs a concrete `spec_json` structure that can support:

1. uploaded source packages
2. Git-based source checkout later
3. Buildpacks as the first builder strategy
4. registry publication
5. release creation and activation

This draft defines the recommended `source_build` payload shape inside `OperationJob.spec_json`.

It is intentionally compatible with the current normalized contract style rather than replacing it.

## Decisions

### 1. `spec_json` keeps one stable top-level shape

For source-based deployment, the operation spec should keep the same top-level pattern as other lifecycle operations.

Recommended top-level fields are:

| Field | Purpose |
| --- | --- |
| `source` | trigger source such as `manualops`, `gitops`, or `store` |
| `adapter` | ingress adapter key |
| `mode` | high-level operation input mode such as `manual-compose` or `source-build` |
| `project_dir` | execution workspace root when already known |
| `compose_project_name` | runtime isolation and compose project identity |
| `exposure_intent` | normalized publication intent when present |
| `metadata` | normalized auxiliary metadata including `runtime_inputs` |
| `source_build` | source-build-specific normalized payload |

This keeps `source_build` as one explicit section without forcing a new top-level schema for all operation types.

### 2. `source_build` is required only when `mode = source-build`

If `spec_json.mode = source-build`, the payload must contain a `source_build` object.

If `mode` is another value, `source_build` should be absent.

This gives the worker and pipeline runner a clear branching point without introducing an ambiguous partially-filled schema.

### 3. Recommended `source_build` shape

`source_build` should use the following normalized structure.

| Field | Required | Purpose |
| --- | --- | --- |
| `source_kind` | yes | `git`, `uploaded-package`, or future normalized kinds |
| `source_ref` | yes | immutable or dereferenceable source identity |
| `source_revision` | no | commit SHA, tag, or package digest when known |
| `workspace_ref` | no | hydrated workspace reference after upload or checkout normalization |
| `builder_strategy` | yes | `buildpacks` initially, extensible later |
| `builder_inputs` | no | normalized non-secret builder options |
| `artifact_publication` | yes | artifact output and publication contract |
| `deploy_inputs` | no | deploy-time inputs required after build |
| `release_metadata` | no | release-facing labels and attribution |

### 4. `artifact_publication` is a nested object, not a flat string set

Registry publication and artifact output should be explicit because source build may later support:

1. local-only image production
2. push to default registry
3. push to explicit remote registry
4. build without immediate publish in selected flows

Recommended shape:

| Field | Required | Purpose |
| --- | --- | --- |
| `mode` | yes | `local-only`, `push`, or future values |
| `target_ref` | no | normalized registry target or repository reference |
| `credential_ref` | no | secret-backed registry credential reference |
| `image_name` | yes | logical image repository name |
| `image_tag` | no | desired tag if not generated by policy |
| `expected_artifact_kind` | yes | `oci-image` for the first implementation |

### 5. `deploy_inputs` wraps runtime deployment inputs after build

`deploy_inputs` should contain only normalized data required after artifact creation.

Recommended first version:

| Field | Required | Purpose |
| --- | --- | --- |
| `runtime_inputs` | no | normalized env and file-like runtime inputs |
| `render_strategy` | no | how post-build runtime config is rendered |
| `activation_policy` | no | release activation preference such as `immediate` |

This keeps build-time concerns separate from deploy-time concerns while preserving one operation spec.

### 6. `release_metadata` is release-facing only

`release_metadata` should carry app-facing or release-facing descriptors, not execution state.

Recommended fields:

| Field | Required | Purpose |
| --- | --- | --- |
| `version_label` | no | operator-facing release label |
| `source_label` | no | human-readable source label |
| `change_summary` | no | concise release note |
| `requested_rollout_reason` | no | optional operator rationale |

Runtime progress, logs, retries, and node status must stay outside `release_metadata`.

### 7. Secret material must not appear inline in `source_build`

The `source_build` object must contain only normalized references for sensitive material.

Examples:

1. source access token -> `secretref://...`
2. registry password -> `secretref://...`
3. deploy-time sensitive env -> represented through normalized runtime input references

Raw tokens, passwords, and private keys must not be stored inline in `spec_json.source_build`.

### 8. Compatibility with current `metadata.runtime_inputs`

Current implementation already places normalized `runtime_inputs` under `spec_json.metadata.runtime_inputs`.

To avoid a disruptive rewrite, the first source-build implementation may keep that behavior while also using `source_build.deploy_inputs.runtime_inputs` as the forward-looking canonical location.

Compatibility rule:

1. writer may mirror normalized `runtime_inputs` into both locations during migration
2. new source-build-aware code should prefer `source_build.deploy_inputs.runtime_inputs`
3. generic legacy paths may continue reading `metadata.runtime_inputs` until the migration is complete

### 9. Canonical example

Recommended example payload:

```json
{
  "source": "manualops",
  "adapter": "source-package",
  "mode": "source-build",
  "project_dir": "/appos/data/apps/operations/my-app",
  "compose_project_name": "my-app",
  "exposure_intent": {
    "domains": ["app.example.com"]
  },
  "metadata": {
    "runtime_inputs": {
      "env": [
        {
          "name": "APP_ENV",
          "kind": "shared-import",
          "set_id": "envset_123",
          "var_id": "envvar_456"
        }
      ],
      "files": [
        {
          "kind": "source-package",
          "name": "app.tar.gz",
          "source_path": "./src/app.tar.gz",
          "uploaded": true
        }
      ]
    }
  },
  "source_build": {
    "source_kind": "uploaded-package",
    "source_ref": "upload://app.tar.gz",
    "source_revision": null,
    "workspace_ref": "workspace://operations/my-app/source",
    "builder_strategy": "buildpacks",
    "builder_inputs": {
      "builder_image": "paketobuildpacks/builder-jammy-base",
      "build_env": {
        "BP_NODE_RUN_SCRIPTS": "build"
      }
    },
    "artifact_publication": {
      "mode": "push",
      "target_ref": "registry://default/apps/my-app",
      "credential_ref": "secretref://registry/default",
      "image_name": "apps/my-app",
      "image_tag": "main-20260401",
      "expected_artifact_kind": "oci-image"
    },
    "deploy_inputs": {
      "runtime_inputs": {
        "env": [
          {
            "name": "APP_ENV",
            "kind": "shared-import",
            "set_id": "envset_123",
            "var_id": "envvar_456"
          }
        ],
        "files": [
          {
            "kind": "source-package",
            "name": "app.tar.gz",
            "source_path": "./src/app.tar.gz",
            "uploaded": true
          }
        ]
      },
      "activation_policy": "immediate"
    },
    "release_metadata": {
      "version_label": "main-20260401",
      "source_label": "uploaded source package",
      "change_summary": "Initial source-based deployment"
    }
  }
}
```

### 10. Validation rules for MVP

The first implementation should validate at least the following:

1. `mode = source-build` requires `source_build`
2. `source_build.source_kind` is required
3. `source_build.source_ref` is required
4. `source_build.builder_strategy` is required
5. `source_build.artifact_publication.mode` is required
6. `source_build.artifact_publication.image_name` is required
7. secret-bearing values must be references, not raw secrets
8. if `artifact_publication.mode = push`, `target_ref` is required
9. if `source_kind = uploaded-package`, at least one uploaded package reference or hydrated workspace reference must exist

## Consequences

This draft gives AppOS a stable path to implement source-based deployment without replacing the current lifecycle operation contract.

The practical impact is:

1. route adapters can normalize source-build input into one explicit `source_build` block
2. worker and pipeline logic can branch on `mode = source-build`
3. runtime-input migration can remain compatible with current `metadata.runtime_inputs`
4. `ReleaseSnapshot` creation can use `source_build` as the canonical source of build and artifact attribution
5. future builder strategies can be added by extending `builder_strategy` and `builder_inputs`, not by redesigning the whole spec
